## 背景

这个博客最初的搜索实现很直接：Astro 在构建搜索页时取出全部已发布文章，把每篇文章渲染成一个 `<article>`，同时将标题、摘要、分类和标签拼成 `data-search-text`。用户每输入一个字符，浏览器便遍历全部文章节点，通过 `includes()` 决定是否隐藏。

文章较少时，这种方案简单、可靠，也不需要额外依赖。但它有两个随内容数量线性增长的成本：

1. 搜索页 HTML 包含全部文章卡片和重复的搜索文本；
2. 每次输入都会遍历并修改大量 DOM 节点。

此外，整句包含匹配对输入格式十分敏感。`Fourier Note` 无法匹配 `Fourier Optics Note`，普通空格无法匹配标题中的连续空格，`thin lens` 与 `thin-lens`、`Goodman's` 与 `Goodman’s` 也会被视为不同内容。

这次改造的目标不是引入完整的站内全文检索，而是在保留“标题、摘要、分类和标签”范围的前提下，让搜索页的初始成本不再依赖文章数量，并补齐这些常见匹配场景。

## 为什么没有采用 Pagefind

Pagefind 很适合为静态站点建立正文全文索引，但它需要在站点构建完成后再次扫描产物，并生成额外的索引文件和运行时代码。对于需要搜索正文、标题权重和大规模内容分片的网站，这笔成本是合理的。

当前博客只需要搜索少量 frontmatter 元数据，而且部署流程已经包含内容校验、图片优化、Astro 类型检查、生产构建和多视口 E2E。为了避免再增加一个 postbuild 阶段，这次选择使用 Astro 自己生成静态 JSON：

```text
Markdown frontmatter
        ↓ Astro build
/search-index.json
        ↓ 首次非空查询时 fetch
规范化的内存索引
        ↓ 输入查询
筛选结果 → 每批渲染 20 条
```

这不会增加新的运行时依赖，也不会扫描文章正文。生成 JSON 时复用现有的 `getAllPosts()`，因此草稿过滤和日期排序仍只有一份规则。

## 静态索引与懒加载

索引只包含结果展示和匹配需要的字段：

```json
{
  "posts": [{
    "title": "Fourier Optics Note-4",
    "description": "Thin-lens phase...",
    "categories": ["Notes"],
    "tags": ["Optics"],
    "dateLabel": "2026/07/17",
    "href": "/blog/fourieroptics4/"
  }]
}
```

搜索页初次打开时不请求这个文件，也不展示文章列表。只有出现非空查询时才加载索引，并在当前页面生命周期内复用同一个 Promise。输入清空后恢复提示状态，无须重新请求。

输入事件增加了 150 ms debounce，避免快速打字时反复筛选。异步搜索还维护一个递增的 revision：索引请求返回后，只有仍对应最新输入的任务可以更新页面，从而避免较早查询覆盖较新查询。

结果使用 DOM API 和 `textContent` 创建，而不是拼接 `innerHTML`。这样即使标题中包含类似 HTML 的文本，也只会作为普通文字显示。命中结果首次只渲染 20 条，用户点击 `Load more` 后再追加下一批，初始 DOM 不再随总命中数增长。

## 匹配规则

索引载入后，每篇文章的四类元数据会被组合并规范化一次。查询采用相同流程：

1. 使用 Unicode NFKC 统一全角和兼容字符；
2. 按 `zh-CN` 规则转换为小写；
3. 将连字符、直引号、弯引号及常见标点转换为空格；
4. 折叠连续空白并拆分词项；
5. 要求查询中的每个词项都能在文章元数据中找到。

词项不需要连续出现，也不需要来自同一字段。因此 `Fourier Note` 可以跨过标题中的 `Optics`，分类和标签也可以共同参与匹配。每个词项仍使用子串判断，所以 `optic` 可以匹配 `optics`。

地址栏中的 `q` 使用稍保守的规范化：清理首尾空格、折叠连续空白并转为小写，但保留用户输入的标点。更新 URL 时使用 `replaceState`，不会为每次按键制造历史记录，也不会丢失其他查询参数和 hash。带 `?q=` 的深链接会在页面打开后立即执行。

## 测试分层

搜索逻辑被提取为无 DOM 的小型模块，Unicode、中文、多关键词、连续空格、标点、部分词匹配和 20 条分页都由 Node 单元测试覆盖。这类测试运行很快，不必为每个输入组合启动浏览器。

桌面 E2E 负责验证真实数据流：

- 空查询不下载索引；
- 标题、摘要、分类和标签均可命中；
- 深链接和 URL 状态正确；
- 无结果与索引请求失败有明确提示；
- 拦截索引并注入 45 条数据，验证 20、40、45 条的加载过程；
- 含 HTML 形式的标题不会被当作节点执行。

移动端和平板不再重复所有功能边界，只保留带 `@responsive` 标记的布局、主题和无障碍冒烟测试。桌面仍执行完整测试集。这让新增覆盖没有等比例放大部署时间。

## 实测结果

测试在同一台 Windows 开发环境、相同暖缓存条件下进行。构建时间各运行三次并取中位数。

| 指标 | 改造前 | 改造后 | 变化 |
|---|---:|---:|---:|
| `npm run build` 中位数 | 8.06 s | 8.56 s | +6.2% |
| E2E runner 时间 | 41.3 s | 29.1–33.1 s | -19.9% 至 -29.5% |
| 搜索页 HTML | 7,424 B | 6,487 B | -12.6% |
| 懒加载搜索索引 | 无 | 2,371 B | 首次搜索时加载 |
| Node 测试数量 | 21 | 24 | +3 |
| E2E 测试数量 | 27 | 27 | 覆盖重新分层 |

构建耗时增加仍低于预设的 10% 控制线，并且没有新增构建阶段。索引会随文章数量增长，但它不再阻塞搜索页初始 HTML，也不会在空查询时下载。最终 E2E 在两次完整运行中分别为 33.1 秒和 29.1 秒；虽然增加了七个搜索场景，通过减少移动和平板的重复执行，总时间仍然下降。

## 当前边界

这个方案仍是客户端元数据搜索，不是通用全文检索。浏览器首次搜索时需要下载完整元数据索引，筛选复杂度仍与文章数量线性相关；`Load more` 只控制 DOM 渲染量，并不会把索引拆成多个网络分片。

对于个人博客和数百到数千篇短元数据记录，这种取舍足够轻量。如果以后出现以下需求，就应该重新评估 Pagefind 或其他预构建倒排索引：

- 需要检索 Markdown 正文或代码块；
- 需要相关性权重、模糊搜索或搜索高亮；
- 索引体积已经明显影响首次查询；
- 客户端线性筛选出现可测量的输入延迟。

可扩展性并不总意味着立即采用最复杂的索引。先把初始 HTML、DOM 更新和重复测试从文章数量中解耦，同时为未来迁移保留清晰边界，是当前规模下更合适的工程选择。
